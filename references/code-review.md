# 阶段 4：Code Review

本文件定义代码审查的检查清单和报告格式。Review 在所有产物生成完成后自动执行，无需用户手动触发。

---

## 4.1 Review 分级标准

```
🔴 Block     — 必须修复才能提交
               影响：安全漏洞、数据丢失、逻辑错误、服务崩溃
               行动：列出问题 + 给出修复代码

🟡 Warning   — 强烈建议修复，不阻塞提交但存在风险
               影响：性能问题、缺失错误处理、潜在竞态条件
               行动：列出问题 + 给出修复建议

🔵 Suggestion — 可选优化，不影响功能
               影响：可读性、风格一致性、扩展性、测试覆盖率
               行动：列出建议，不强制要求
```

**有任何 🔴 Block 时**，必须在报告末尾输出：
> ⛔ **发现 [N] 个 Block 级问题，请修复后再提交。**

---

## 4.2 安全检查清单（Block 级）

### SQL 注入
- [ ] 所有 SQL 查询使用参数化查询，无字符串拼接 SQL
- [ ] ORM 的 `createQueryBuilder` 中无直接插入用户输入

### 未鉴权接口
- [ ] 所有需要登录的接口有鉴权中间件/装饰器
- [ ] 涉及用户数据的接口有数据归属校验（不能查看他人数据）
- [ ] 管理员接口有角色权限校验

### 敏感信息泄露
- [ ] 错误响应不返回数据库异常堆栈
- [ ] 日志不记录密码、Token、身份证号等敏感字段
- [ ] API 响应不包含不必要的内部字段（如密码 hash）

### 输入校验
- [ ] 所有外部输入有长度限制
- [ ] 文件上传有类型和大小校验
- [ ] 数值类型有合理范围限制

### 其他安全
- [ ] 无硬编码的密钥、密码、Token
- [ ] 外部 URL 跳转有白名单校验（防开放重定向）
- [ ] 批量操作有数量上限（防止 DoS）

---

## 4.3 性能检查清单（Warning 级）

### 数据库
- [ ] 新查询条件字段有对应索引（对照 Migration 中的索引）
- [ ] 无 N+1 查询（循环中调用 DB）
- [ ] 列表查询有分页，无 `SELECT *` 全表扫描
- [ ] 大数据量操作使用批量处理（不一次性加载所有数据）
- [ ] 事务粒度合理，不在事务中做耗时操作（如外部 HTTP 请求）

### 并发
- [ ] 共享状态的修改有加锁或乐观锁
- [ ] 重复提交场景有幂等处理（数据库唯一约束或业务去重）
- [ ] 定时任务有分布式锁（多实例部署场景）

### 缓存
- [ ] 高频查询是否考虑缓存（对照项目现有缓存使用方式）
- [ ] 缓存有合理的过期时间
- [ ] 缓存更新策略（写穿 / 失效 / 写回）是否合适

---

## 4.4 代码质量检查清单（Warning / Suggestion）

### 错误处理
- [ ] 所有异常情况有明确的错误处理（不静默吞掉 catch）
- [ ] 错误信息对用户友好，不暴露技术细节
- [ ] 异步操作有错误处理（不遗漏 await 导致 Unhandled Promise Rejection）

### 代码风格一致性（对照 Project Profile）
- [ ] 命名规范与项目一致
- [ ] 注释风格与项目一致
- [ ] 响应格式与项目一致
- [ ] 日志调用方式与项目一致

### 可维护性
- [ ] 单个函数不超过 50 行（超过则建议拆分）
- [ ] 无魔法数字（应提取为常量并注释含义）
- [ ] 无重复代码（超过 3 处相似逻辑应提取函数）
- [ ] 依赖注入而非硬编码依赖（便于单元测试）

### 测试质量
- [ ] 核心业务逻辑有测试覆盖
- [ ] 测试用例包含异常路径
- [ ] 测试无依赖执行顺序（每个测试独立可运行）
- [ ] Mock 的范围合理（不过度 Mock）

## 4.5 Code Review 报告格式

```markdown
# Code Review 报告

## 总体评估

| 维度 | 评级 | 说明 |
|-----|-----|-----|
| 安全性 | ✅ 通过 / ⚠️ 警告 / ❌ 阻塞 | — |
| 性能 | ✅ / ⚠️ / ❌ | — |
| 代码质量 | ✅ / ⚠️ / ❌ | — |
| 风格一致性 | ✅ / ⚠️ / ❌ | — |
| 测试覆盖 | ✅ / ⚠️ / ❌ | — |

---

## 🔴 Block 问题（必须修复）

### [Block-1] SQL 注入风险
**位置：** `src/repositories/user.repository.ts:42`
**问题：** 用户输入直接拼接到 SQL 字符串
**修复：**
```typescript
// 修复前
const users = await query(`SELECT * FROM users WHERE name = '${name}'`);

// 修复后
const users = await query('SELECT * FROM users WHERE name = $1', [name]);
```

---

## 🟡 Warning 问题（建议修复）

### [Warning-1] 缺少索引导致全表扫描
**位置：** `src/migrations/xxx-AddOrderTable.ts`
**问题：** `orders.user_id` 字段无索引，查询用户订单列表将全表扫描
**建议：** 在 Migration 中添加 `CREATE INDEX idx_orders_user_id ON orders(user_id)`

### [Warning-2] 事务中包含外部 HTTP 请求
**位置：** `src/services/order.service.ts:78`
**问题：** 在数据库事务内调用了外部支付接口，若支付接口超时将导致事务长时间占用连接
**建议：** 将外部 HTTP 调用移到事务之外

---

## 🔵 Suggestion（可选优化）

### [Suggestion-1] 可提取公共常量
**位置：** `src/services/order.service.ts:15, 67, 89`
**建议：** `'pending'` 出现 3 次，建议提取为 `ORDER_STATUS.PENDING` 常量

### [Suggestion-2] 测试覆盖率可增强
**建议：** 可补充"库存不足时创建订单失败"的测试用例

---

## 总结

- Block：1 个
- Warning：2 个
- Suggestion：2 个

⛔ **发现 1 个 Block 级问题，请修复后再提交。**
```

## 4.6 Review 后的行动

Review 报告输出后，询问用户：

```
Code Review 完成。是否需要：
A. 我直接帮你修复 Block 和 Warning 问题
B. 只修复 Block 问题
C. 查看某个具体问题的详细修复方案
D. 确认已了解，不需要修复
```

如果用户选择 A 或 B，按照 `references/development.md` 中的 diff 格式输出修复方案，并重新执行 Review 确认修复有效。
